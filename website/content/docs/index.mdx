OpenSpec helps you and your AI coding assistant **agree on what to build before any code is written.**

You describe a change. The AI drafts a short spec and a task list. You both look at the same plan. Then the work happens. No more discovering halfway through that the AI built something you did not ask for.

That is the whole pitch. The rest of these docs is detail.

<Callout title="The one habit worth forming">
When you are not sure what to build, start with `/opsx:explore`. It is a no-stakes thinking partner that reads your code, weighs options, and sharpens a fuzzy idea into a concrete plan before anything is written. See [Explore First](/docs/explore).
</Callout>

## The problem it solves

You ask your AI assistant for a feature. It confidently writes 300 lines. Halfway through review you realize it solved a slightly different problem — it cached the wrong layer, or touched auth when it should not have, or built the theme toggle but skipped the system-preference case you cared about most. Now you are reverse-engineering its intent from a diff, inside a chat thread that scrolls away tomorrow.

That failure mode has one root cause: **the requirements lived only in chat.** Nothing wrote down what "done" meant before the code got written.

OpenSpec writes it down. You state intent in plain language, the AI drafts the details, and you both approve a short plan *before* code happens. The plan and the code then live side by side in your repo — so six months later, the spec still explains why the system works the way it does, to you and to the next AI session alike.

The whole bet is that catching a wrong turn in a one-paragraph plan is nearly free, while catching it in 300 lines of code is not.

## The thirty-second version

```text
1. Install     npm install -g @fission-ai/openspec@latest
2. Initialize  cd your-project && openspec init
3. Explore     (in your AI chat)  /opsx:explore          ← optional, great habit
4. Propose     (in your AI chat)  /opsx:propose add-dark-mode
5. Build       (in your AI chat)  /opsx:apply
6. Archive     (in your AI chat)  /opsx:archive
```

Steps 1 and 2 run in your **terminal**. The rest run in your **AI assistant's chat**. That split is the one thing worth memorizing, and [How Commands Work](/docs/how-commands-work) explains exactly why.

## How it compares

There are heavier ways to do spec-driven development, and lighter ways (none at all). OpenSpec aims for the honest middle.

| | The approach | The catch |
|---|---|---|
| **Spec Kit** (GitHub) | Thorough, structured | Rigid phase gates, lots of Markdown, Python setup |
| **Kiro** (AWS) | Powerful, integrated | Locked into their IDE and a limited set of models |
| **No specs at all** | Zero overhead | Vague prompts, unpredictable results, no record of *why* |
| **OpenSpec** | Lightweight, fluid, in your repo | Adds one small step — worth it whenever agreement matters |

OpenSpec works with the AI tools you already use, fits existing codebases, and never makes you finish planning before you are allowed to build.

## A philosophy in four lines

```text
fluid not rigid          — work on what makes sense, in any order
iterative not waterfall  — learn as you build, refine as you go
easy not complex         — initialize in seconds, customize only if you need to
brownfield-first         — built for existing systems, not just green fields
```

<Callout title="When the ceremony isn't worth it">
OpenSpec is lightweight, but it is not free — it adds a step. For a truly trivial one-line fix, skip it; that is the honest call. Use OpenSpec where agreement matters, which turns out to be most of the time once you are working with an AI that will confidently build whatever you vaguely asked for.
</Callout>

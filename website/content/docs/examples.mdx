Real changes, start to finish. Each recipe shows the commands you would type and what you would see back. Match your situation to a pattern and copy it.

<Callout title="Reading the transcripts">
`You:` and `AI:` lines happen in your **AI assistant's chat**. Lines starting with `$` happen in your **terminal**. New to that split? See [How Commands Work](/docs/how-commands-work).
</Callout>

## Recipe 1 — A small feature, the fast path

**When:** you know what you want and it is a contained piece of work. The most common recipe. Three commands: propose, build, archive.

```text
You: /opsx:propose add-logout-button

AI:  Created openspec/changes/add-logout-button/
     ✓ proposal.md — why and what
     ✓ specs/      — the new requirement and its scenarios
     ✓ design.md   — technical approach
     ✓ tasks.md    — implementation checklist
     Ready for implementation. Run /opsx:apply.
```

Now read the plan. This is the moment OpenSpec is built for: catching a wrong assumption while it is still one paragraph, not 400 lines of code. Edit any artifact directly if something is off, then continue.

```text
You: /opsx:apply

AI:  Working through tasks...
     ✓ 1.1 Add logout button to header
     ✓ 1.2 Clear session on click
     ✓ 1.3 Add confirmation dialog
     All tasks complete!

You: /opsx:archive

AI:  ✓ Merged specs into openspec/specs/auth/spec.md
     ✓ Moved to openspec/changes/archive/2026-06-22-add-logout-button/
     Done. Ready for the next change.
```

The logout behavior is now part of your specs, filed away with its full context.

## Recipe 2 — A bug fix

**When:** something is broken and you want the fix recorded as a deliberate change to behavior, not a mystery commit.

Bug fixes work exactly like features. The trick is to frame the proposal around the *correct* behavior, not "fix the bug."

```text
You: /opsx:propose fix-login-redirect-loop

AI:  Created openspec/changes/fix-login-redirect-loop/
     The proposal captures the current broken behavior and the
     intended behavior. The delta spec MODIFIES the redirect
     requirement so it's now precise and testable.
```

Because the fix lands as a `MODIFIED` requirement with a fresh scenario, the next person — or the next AI session — sees not just that you fixed it, but what "correct" means.

<Callout title="Tip">
A good bug-fix scenario is the regression test in prose: "GIVEN a logged-out user, WHEN they submit valid credentials, THEN they land on the dashboard and are not redirected again." Write that, and the implementation has a clear target.
</Callout>

## Recipe 3 — Explore before you commit

**When:** you have a problem but not yet a plan. Start with `/opsx:explore` — a thinking partner that creates no artifacts and writes no code.

```text
You: /opsx:explore

AI:  What would you like to explore?

You: Page loads feel slow but I don't know where the time goes.

AI:  Let me look... I found three likely culprits:
     1. Large unoptimized images on the product list
     2. Synchronous data fetching that blocks render
     3. Re-renders triggered by context changes
     Which should we tackle first?

You: The data fetching one.

You: /opsx:propose optimize-product-list-fetching

AI:  Created the change, with a proposal informed by what we just
     found. Ready for implementation.
```

Exploration clarifies your thinking *before* you spend a change on it. [More on Explore First.](/docs/explore)

## Recipe 4 — Juggling two changes at once

**When:** you are mid-feature and an urgent fix jumps the queue. Changes are independent folders, so parallel work does not conflict.

```text
You: I'm partway through add-dark-mode, but I need to fix a login bug now.

You: /opsx:propose fix-login-redirect

AI:  Created openspec/changes/fix-login-redirect/. Ready for implementation.

You: /opsx:apply
AI:  Implemented the fix. All tasks complete.

You: /opsx:archive
AI:  ✓ Archived fix-login-redirect.

You: Back to dark mode.

You: /opsx:apply add-dark-mode
AI:  Resuming add-dark-mode. Picking up at task 2.3: Update Header...
```

Naming the change in `/opsx:apply add-dark-mode` points the AI at a specific change when more than one is active. Because tasks track completion in `tasks.md`, the AI knows exactly where you stopped.

## Recipe 5 — A refactor with no behavior change

**When:** you are restructuring code, and externally visible behavior should stay identical.

This is the interesting case: a pure refactor has *nothing to add to your specs*. The behavior contract does not change, only the implementation. So the work lives in design and tasks, and the spec delta is empty.

```text
You: /opsx:propose refactor-payment-module

AI:  Created the change. The proposal states the goal (split the
     payment module, no behavior change) and the design captures
     the new structure. No spec changes, since behavior is identical.
```

When you archive a change that does not touch specs, tell the terminal command to skip the spec step:

```bash
$ openspec archive refactor-payment-module --skip-specs
```

The principle: specs describe behavior, so if behavior did not change, the spec should not either.

## Recipe 6 — Step by step, or hands-on

**When:** a complex change where you want to review each artifact before moving on, or you simply want to *feel* the workflow on your own code. Both use the [expanded profile](/docs/the-workflow):

```bash
$ openspec config profile      # select the expanded workflows
$ openspec update              # apply them to this project
```

Then build incrementally with `/opsx:new` and `/opsx:continue` (review each artifact as it lands), or run `/opsx:onboard` for a guided 15–30 minute tour that finds a small, safe improvement in your codebase and walks you through the entire loop — leaving you with a real change you can keep or discard.

## Check your work any time

From your terminal, inspect the state of things without changing anything:

```bash
$ openspec list                   # active changes
$ openspec show add-dark-mode     # one change in detail
$ openspec validate add-dark-mode # check structure
$ openspec view                   # interactive dashboard
```

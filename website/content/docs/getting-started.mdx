This guide takes you from nothing to your first shipped change. It assumes you have [installed the CLI](/docs/installation).

<Callout title="Where do these commands run?">
Two places, and mixing them up is the most common early stumble.

- `openspec ...` commands (like `openspec init`) run in your **terminal**.
- `/opsx:...` commands (like `/opsx:propose`) run in your **AI assistant's chat** — the same box where you ask it to write code.

There is no separate "interactive mode" to launch. You type the slash command in chat and your assistant takes it from there. Full explanation: [How Commands Work](/docs/how-commands-work).
</Callout>

## Your first five minutes

```text
TERMINAL   $ npm install -g @fission-ai/openspec@latest
TERMINAL   $ cd your-project && openspec init
AI CHAT      /opsx:explore                 (optional: think it through first)
AI CHAT      /opsx:propose add-dark-mode   (AI drafts the plan; you review it)
AI CHAT      /opsx:apply                   (AI builds it)
AI CHAT      /opsx:archive                 (specs updated, change filed away)
```

Two terminal steps to set up, then you live in chat.

## Initialize your project

In your project directory, run:

```bash
openspec init
```

This creates an `openspec/` folder and generates instructions for your AI tool so it knows the OpenSpec slash commands. You only do this once per project.

Here is what `openspec init` sets up:

```text
openspec/
├── specs/         # Source of truth — how your system behaves today
│   └── <domain>/
│       └── spec.md
├── changes/       # Proposed updates — one folder per change
│   └── <change-name>/
│       ├── proposal.md
│       ├── design.md
│       ├── tasks.md
│       └── specs/   # Delta specs — what is changing
└── config.yaml    # Optional project configuration
```

Two folders matter most. **`specs/`** is what is true today. **`changes/`** is what you are proposing. When a change is done, its edits merge into `specs/`. That is the entire loop.

## Ship your first change

Let's add dark mode to an app.

<Steps>

<Step>

### Propose the change

In your AI chat, type:

```text
/opsx:propose add-dark-mode
```

The AI creates a change folder and drafts everything in it:

```text
Created openspec/changes/add-dark-mode/
  ✓ proposal.md — why we are doing this, what is changing
  ✓ specs/       — requirements and scenarios
  ✓ design.md    — technical approach
  ✓ tasks.md     — implementation checklist
Ready for implementation!
```

**This is the moment to slow down.** Read the proposal. Skim the spec deltas. If the scope is wrong or the approach is off, say so now — fixing a one-paragraph plan is free. Fixing 400 lines of code later is not.

</Step>

<Step>

### Build it

Once the plan looks right:

```text
/opsx:apply
```

The AI works through the task list, writing code and checking off each item:

```text
Working through tasks...
  ✓ 1.1 Created ThemeContext with light/dark state
  ✓ 1.2 Added CSS custom properties to globals.css
  ✓ 1.3 Implemented localStorage persistence
  ✓ 2.1 Created ThemeToggle component
  ...
All tasks complete!
```

If you discover the design needs a tweak mid-build, just update the artifact and keep going. Nothing is locked.

</Step>

<Step>

### Archive it

When the work is done and verified:

```text
/opsx:archive
```

```text
Archiving add-dark-mode...
  ✓ Merged specs into openspec/specs/ui/spec.md
  ✓ Moved to openspec/changes/archive/2025-01-24-add-dark-mode/
Done! Ready for the next feature.
```

Your spec deltas are now part of the main specs. They document how your system works — for you, for your team, and for the next AI session.

</Step>

</Steps>

## Check on your work from the terminal

```bash
openspec list                  # List active changes
openspec show add-dark-mode    # View a change in detail
openspec validate add-dark-mode  # Check spec formatting
openspec view                  # Interactive dashboard
```

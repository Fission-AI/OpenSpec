## The core nouns

**Spec.** A document describing how part of your system behaves. Specs live in `openspec/specs/`, are organized by domain, and are made of requirements and scenarios. The agreed-upon answer to "what does this software do?"

**Source of truth.** The `openspec/specs/` directory as a whole — the current, agreed-upon behavior of your system. Changes propose edits to it; archiving applies them.

**Change.** One unit of work, packaged as a folder under `openspec/changes/<name>/`. It holds everything about that work: the proposal, design, tasks, and the spec edits it introduces. One change, one feature or fix.

**Artifact.** A document inside a change — the proposal, the delta specs, the design, or the tasks. Created in dependency order, each feeding the next.

**Delta spec.** A spec inside a change that describes only what is changing, using `ADDED`, `MODIFIED`, and `REMOVED` sections, rather than restating the whole spec. This is what lets OpenSpec edit existing systems cleanly.

**Domain.** A logical grouping for specs, like `auth/`, `payments/`, or `ui/`. You choose domains that match how you think about your system.

## Inside a spec

**Requirement.** A single behavior the system must have, usually written with an RFC 2119 keyword: "The system SHALL expire sessions after 30 minutes." The *what*, not the *how*.

**Scenario.** A concrete, testable example of a requirement in action, typically in Given/When/Then form. You could write an automated test from one.

**RFC 2119 keywords.** MUST, SHALL, SHOULD, and MAY — words with standardized meaning about how strict a requirement is. MUST and SHALL are absolute; SHOULD is recommended with room for exceptions; MAY is optional.

## The artifacts

**Proposal (`proposal.md`).** The *why* and *what*: intent, scope, and high-level approach. The first artifact you create.

**Design (`design.md`).** The *how*: technical approach, architecture decisions, files you expect to touch. Optional for simple changes.

**Tasks (`tasks.md`).** The implementation checklist, with checkboxes. The AI works through it during `/opsx:apply`.

## The lifecycle

**Archive.** Finishing a change. Its delta specs merge into the main specs, and the folder moves to `openspec/changes/archive/YYYY-MM-DD-<name>/`.

**Sync.** Merging a change's delta specs into the main specs *without* archiving. Usually automatic (archive offers it), but available on its own as `/opsx:sync`.

## Workflow and commands

**OPSX.** The current OpenSpec workflow, built around fluid actions instead of rigid phases. Its slash commands all start with `/opsx:`.

**Slash command.** A command you type into your AI assistant's chat, like `/opsx:propose`. Drives the workflow. Not a terminal command.

**CLI.** The `openspec` program you run in your terminal. Sets up projects, lists and validates changes, opens the dashboard, archives. The terminal half of OpenSpec.

**Skill.** A folder of instructions (`.../skills/openspec-*/SKILL.md`) your AI assistant auto-detects and follows. The emerging cross-tool standard for delivering the workflow.

**Profile.** The set of slash commands installed in your project. **Core** (default) is `propose`, `explore`, `apply`, `sync`, `archive`. The **expanded** set adds `new`, `continue`, `ff`, `verify`, `bulk-archive`, `onboard`.

## Customization

**Schema.** The definition of which artifacts a workflow has and how they depend on one another. The built-in default is `spec-driven` (proposal → specs → design → tasks). Fork it or write your own.

**Template.** A Markdown file inside a schema that shapes what the AI generates for an artifact. Edit it and the AI's output changes immediately.

**Project config (`openspec/config.yaml`).** Per-project settings: the default schema, the `context:` injected into every planning request, and per-artifact `rules:`.

**Context injection.** Putting project background in `config.yaml`'s `context:` so it is automatically added to every artifact the AI generates.

**Enablers, not gates.** The principle that artifact dependencies show what becomes *possible* next, not what is *required* next. You can revisit and edit any artifact at any time.

## Coordination across repos (beta)

These apply only if your planning spans more than one repo. See [Stores](/docs/stores).

**Store.** A standalone repo whose whole job is planning. The same `openspec/` shape (specs and changes) plus a small identity file. Registered on your machine once, by name.

**Reference.** A declaration, in a code repo's `openspec/config.yaml`, of a store that repo draws on, read-only. The repo keeps its own root; `openspec instructions` gains an index of the referenced store's specs.

**Working context.** What `openspec context` assembles for the current repo: its OpenSpec root plus every store it references. The answer to "what am I working with?"

**Workset.** A personal, machine-local set of folders you open together. Created with `openspec workset create`; nothing about those local paths is committed to shared planning.

The `openspec` CLI runs in your **terminal**. It sets up projects, browses and validates your specs and changes, and archives completed work. (The day-to-day *driving* — proposing and building — happens with [slash commands](/docs/reference/slash-commands) in your AI chat.)

Every command supports `--help`, `--version` / `-V`, and `--no-color`.

## Setup

### `openspec init`

```bash
openspec init [path]
```

Initializes OpenSpec in a project: creates the `openspec/` directory and generates command files for your AI tool. Run once per project.

| Option | Effect |
|--------|--------|
| `--tools <list>` | Generate integrations for specific tools only |
| `--profile <profile>` | Start with a specific profile (e.g. `core`) |
| `--force` | Overwrite existing files |

### `openspec update`

```bash
openspec update [path]
```

Regenerates your AI tool's command files after you upgrade the CLI or change your profile. Run this inside each project so the latest slash commands are active.

## Browse

### `openspec list`

```bash
openspec list [--changes] [--specs] [--json]
```

Lists active changes (or specs). Add `--sort <order>` to control ordering.

### `openspec show`

```bash
openspec show [item-name] [--json]
```

Shows the details of a change or spec — a consolidated view of its artifacts. Handy flags: `--deltas-only`, `--requirements`, `--no-scenarios`, `-r, --requirement <id>`.

### `openspec view`

```bash
openspec view
```

An interactive dashboard for exploring your specs and changes. A viewer, not an editor — you still propose and build with slash commands.

## Validate

### `openspec validate`

```bash
openspec validate [item-name] [--all] [--strict]
```

Checks changes and specs for structural issues. Useful before archiving. Scope it with `--changes`, `--specs`, or `--all`; tighten it with `--strict`.

## Lifecycle

### `openspec archive`

```bash
openspec archive [change-name]
```

Archives a completed change and merges its delta specs into the main specs. Most people trigger this through `/opsx:archive` in chat, but the CLI form is here when you want it.

| Option | Effect |
|--------|--------|
| `-y, --yes` | Skip the confirmation prompt |
| `--skip-specs` | Archive without merging spec deltas |
| `--no-validate` | Skip validation before archiving |

## Configure

### `openspec config`

```bash
openspec config <subcommand>
```

Manage project configuration. Subcommands: `path`, `list`, `get <key>`, `set <key> <value>`, `unset <key>`, `reset`, `edit`, and `profile [preset]`.

```bash
openspec config profile        # interactive profile picker
openspec config profile core   # jump straight to the core preset
```

See [Customization](/docs/customization) for what lives in `config.yaml`.

## More command groups

OpenSpec has additional commands for power users and beta features. Run any of them with `--help` for details.

| Group | Purpose |
|-------|---------|
| `openspec schema …` | Create, fork, validate, and inspect workflow schemas (`init`, `fork`, `validate`, `which`) |
| `openspec store …` | Manage standalone planning repositories (beta): `setup`, `register`, `list`, `doctor`, … |
| `openspec new` / `status` / `instructions` / `templates` / `schemas` | The lower-level, artifact-driven workflow commands the slash commands build on |
| `openspec completion …` | Install shell completions for bash, zsh, fish, or powershell |
| `openspec doctor` | Report health for the resolved root and any referenced stores |

## Utility

### `openspec feedback`

```bash
openspec feedback "your message"
```

Sends feedback straight from your terminal by opening a GitHub issue. Add `--body <text>` for a longer description.

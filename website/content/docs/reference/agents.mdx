The slash commands are a friendly face over a CLI that agents can drive directly. That is the point: OpenSpec is **AI-native**, so a coding agent can plan, build, and archive without a human translating between chat and the terminal.

## Everything speaks JSON

Almost every `openspec` command accepts `--json` and prints structured output instead of prose:

```bash
openspec list --json
openspec show add-dark-mode --json
openspec status --change add-dark-mode --json
openspec validate --all --json
openspec schemas --json
```

An agent uses these to discover state — what changes exist, which artifacts are ready, what failed validation — and to decide what to do next, with no screen-scraping.

## The workflow primitives

The `/opsx:*` skills are built on a small set of lower-level commands. You can call them yourself, in scripts or from an agent:

| Command | What an agent gets |
|---------|--------------------|
| `openspec new change <name>` | Creates the change folder and metadata |
| `openspec status` | Which artifacts exist and which are unblocked next |
| `openspec instructions <artifact>` | The enriched prompt for creating an artifact (with your project `context:` injected) |
| `openspec templates` | The resolved template paths for a schema |
| `openspec schemas` | The available workflow schemas |
| `openspec validate` | Structural problems, by file |
| `openspec archive` | Merge deltas and file the change away |

A typical agent loop: read `status --json`, fetch `instructions` for the next ready artifact, write the file, `validate`, repeat — then `archive`.

## Deterministic by design

Because the engine lives in the CLI — not in any one model's head — the rules are the same everywhere. What a change folder looks like, how artifacts depend on each other, how deltas merge: the CLI enforces all of it, so different agents and different models produce compatible results. The slash command files generated by `openspec init` are just the per-tool wrappers that call this engine.

## The full contract

Exact flags, JSON shapes, and the surfaces agents are expected to drive are documented in the [agent contract](https://github.com/Fission-AI/OpenSpec/blob/main/docs/agent-contract.md) in the repository.

<Callout type="info">
One known wrinkle: store-family JSON keys are `snake_case` while workflow-family keys are `camelCase`. It is documented in the agent contract; unifying it is deferred to a versioned release.
</Callout>

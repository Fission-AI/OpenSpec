OpenSpec works out of the box with zero configuration. When you want it to fit your team more closely, there are two levels: **project config** (easy, do this first) and **custom schemas** (powerful, for when you need a different workflow).

## Project config

Everything lives in `openspec/config.yaml`. The most valuable key is `context:` — anything you put there is injected into every artifact the AI drafts, so every proposal respects your stack and conventions.

```yaml
schema: spec-driven

context: |
  Tech stack: TypeScript, React, Node.js, PostgreSQL
  API style: RESTful, documented in docs/api.md
  Testing: Jest + React Testing Library

rules:
  proposal:
    - Include a rollback plan
  specs:
    - Use Given/When/Then format
```

| Key | What it does |
|-----|--------------|
| `schema` | The default workflow schema (built-in default is `spec-driven`) |
| `context` | Project background injected into all artifacts — your single highest-leverage setting |
| `rules` | Per-artifact guidance, e.g. always add a rollback plan to proposals |

Manage config from the terminal too:

```bash
openspec config list           # show all settings
openspec config get <key>      # read one
openspec config set <key> <value>
openspec config edit           # open in your editor
```

## Custom schemas

A **schema** defines which artifacts a workflow has and how they depend on each other. The built-in `spec-driven` schema gives you `proposal → specs → design → tasks`. If your team works differently — research first, or skip design entirely — you can define your own.

```bash
openspec schema init research-first      # create from scratch
openspec schema fork spec-driven my-flow # copy a built-in one to customize
```

A schema is a YAML file describing artifacts and their dependencies:

```yaml
# openspec/schemas/research-first/schema.yaml
name: research-first
version: 1
description: Do research before proposing, then skip straight to tasks.

artifacts:
  - id: research
    generates: research.md
    requires: []            # do research first

  - id: proposal
    generates: proposal.md
    requires: [research]    # proposal informed by research

  - id: tasks
    generates: tasks.md
    requires: [proposal]    # skip specs/design, go straight to tasks
```

<Callout title="Dependencies are enablers, not gates">
`requires` shows what is *possible* to create next, not what you are *forced* to create. It exists so the AI has the context it needs — for example, you cannot write good tasks without specs to base them on.
</Callout>

Templates for each artifact live in `openspec/schemas/<name>/templates/` as Markdown files. Schemas are version-controlled with your code, so the whole team shares the same workflow.

### How a schema gets chosen

OpenSpec resolves the schema in this order:

1. A `--schema <name>` CLI flag
2. The change's own `.openspec.yaml` metadata
3. `schema:` in `openspec/config.yaml`
4. The default, `spec-driven`

Inspect what is available and where it resolves from:

```bash
openspec schemas            # list available schemas
openspec schema which       # show where the active schema resolves from
openspec schema validate    # check a schema's structure and templates
```

## Community schemas

Third-party schema bundles are distributed as standalone repositories — opinionated workflows that integrate OpenSpec with other tools. They are the OpenSpec equivalent of a plugin catalog. Browse them from the project's customization docs on GitHub.

## Installation and setup

### `openspec: command not found`

The CLI is not installed, or your shell cannot find it.

```bash
npm install -g @fission-ai/openspec@latest
openspec --version
```

If it installed but still is not found, your global npm bin directory is probably not on your `PATH`. Run `npm bin -g` to see where global binaries live, and add that path to your shell profile.

### "Requires Node.js 20.19.0 or higher"

OpenSpec runs on Node 20.19.0+. Check and upgrade if needed:

```bash
node --version
```

If you installed OpenSpec with bun, note it still *runs* on Node, so you need Node 20.19.0+ on your `PATH` regardless.

### `openspec init` didn't configure my AI tool

Init asks which tools to set up. If you skipped yours, run it again, or use the non-interactive form:

```bash
openspec init --tools claude,cursor   # or --tools all / --tools none
```

## Commands don't show up

If `/opsx:propose` (or your tool's equivalent) does not appear, work down this list, fastest first:

<Steps>
<Step>**You may be in the wrong place.** Slash commands go in your AI assistant's chat, not your terminal. See [How Commands Work](/docs/how-commands-work).</Step>
<Step>**Regenerate the files** from your project root: `openspec update`.</Step>
<Step>**Restart your assistant.** Most tools scan for skills and commands at startup.</Step>
<Step>**Confirm the files exist.** For Claude Code, check `.claude/skills/` for `openspec-*` folders. Other tools use their own directories (see [Supported Tools](/docs/reference/supported-tools)).</Step>
<Step>**Check you initialized this project.** Skills are written per project — run `openspec init` (or `openspec update`) in the right folder.</Step>
<Step>**Confirm your tool uses command files.** A few (Kimi CLI, Trae, ForgeCode, Mistral Vibe) use skill-based invocations instead.</Step>
</Steps>

## Working with changes

### "Change not found"

The command could not tell which change you meant. Name it explicitly, or check what exists:

```bash
openspec list                # see active changes
/opsx:apply add-dark-mode    # name the change in chat
```

### "No artifacts ready"

Every artifact is either already created or blocked on a dependency. See what is blocking:

```bash
openspec status --change <name>
```

Then create the missing dependency first. The order: proposal enables specs and design; specs and design together enable tasks.

### `openspec validate` reports warnings or errors

Validation checks structure. Read the message — it names the file and the issue.

```bash
openspec validate <name>           # one item
openspec validate --all            # everything
openspec validate --all --strict   # stricter, good for CI
```

Common causes: a missing required section (like a spec with no scenarios) or a malformed delta header. Fix the file and re-run.

### The AI created incomplete or wrong artifacts

It did not have enough context. A few levers:

- Add project context in `openspec/config.yaml` so your stack and conventions are injected into every request. See [Customization](/docs/customization).
- Add per-artifact `rules:` for guidance that only applies to, say, specs.
- Give a more detailed description when you propose.
- Use the expanded `/opsx:continue` to create one artifact at a time and review each.

### Archive won't finish, or warns about incomplete tasks

Archive will not *block* on incomplete tasks, but it warns you. If tasks remain on purpose, proceed; otherwise finish them first. Archive will also offer to sync your delta specs into the main specs — say yes unless you have a reason not to.

## Configuration

### My `config.yaml` isn't being applied

Three usual suspects:

1. **Wrong filename.** It must be `openspec/config.yaml`, not `.yml`.
2. **Invalid YAML.** Run it through a validator; the CLI also reports syntax errors with line numbers.
3. **You expected a restart.** You do not need one — config changes take effect immediately.

### "Context too large"

The `context:` field is capped at 50 KB, on purpose, because it is injected into every request. Summarize it or link out to longer docs. Lean context also produces better, faster results.

### "Schema not found"

The schema you referenced does not exist. List what is available and check spelling:

```bash
openspec schemas               # list available
openspec schema which <name>   # where a schema resolves from
```

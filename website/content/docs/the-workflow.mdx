OpenSpec's default workflow is short. Most days look like this:

```text
/opsx:explore ──► /opsx:propose ──► /opsx:apply ──► /opsx:archive
  (optional)
```

Start with `/opsx:explore` when you are figuring out what to do. Jump straight to `/opsx:propose` when you already know. Both ship in the default profile, so they are always there.

## The four core moves

<Steps>

<Step>

### Explore — think it through *(optional)*

```text
/opsx:explore
```

A no-stakes thinking partner. It reads your code, weighs options, and turns a fuzzy idea into a concrete plan — all before any artifact exists. This is the single best habit for working with an AI that will otherwise confidently build the wrong thing. [Read the Explore guide.](/docs/explore)

</Step>

<Step>

### Propose — draft the plan

```text
/opsx:propose add-dark-mode
```

In one step, the AI creates the change folder and drafts the proposal, spec deltas, design, and task list. This is your review moment: read it and adjust before code is written.

</Step>

<Step>

### Apply — build it

```text
/opsx:apply
```

The AI works through the task list, writing code and checking off each item. Learn something mid-build? Update the artifact and keep going.

</Step>

<Step>

### Archive — fold it into the truth

```text
/opsx:archive
```

Spec deltas merge into your main specs, and the change moves to `changes/archive/` with a date stamp. (There is also `/opsx:sync`, which merges deltas into specs without archiving; archive will prompt you if a sync is needed, so you rarely call it directly.)

</Step>

</Steps>

## Profiles: core vs expanded

OpenSpec ships two sets of commands. The default is **core** — five commands that cover the whole loop.

<Tabs items={['Core profile (default)', 'Expanded profile']}>
  <Tab value="Core profile (default)">

| Command | What it does |
|---------|--------------|
| `/opsx:explore` | Think through an idea before committing |
| `/opsx:propose` | Create a change and draft all planning artifacts at once |
| `/opsx:apply` | Implement the tasks |
| `/opsx:sync` | Merge delta specs into the main specs |
| `/opsx:archive` | Archive a completed change |

The core profile is all most people ever need.

  </Tab>
  <Tab value="Expanded profile">

The expanded profile adds finer-grained control for bigger or more deliberate work:

| Command | What it does |
|---------|--------------|
| `/opsx:new` | Scaffold a change without drafting artifacts yet |
| `/opsx:continue` | Create the next artifact, one at a time |
| `/opsx:ff` | Fast-forward: create all planning artifacts in order |
| `/opsx:verify` | Check that the implementation matches the artifacts |
| `/opsx:bulk-archive` | Archive several completed changes at once |
| `/opsx:onboard` | A guided tutorial through the full workflow on your own code |

The expanded path looks like:

```text
/opsx:new ──► /opsx:ff or /opsx:continue ──► /opsx:apply ──► /opsx:verify ──► /opsx:archive
```

  </Tab>
</Tabs>

### Switching profiles

The default global profile is `core`. To enable the expanded commands:

```bash
openspec config profile     # interactive picker
# ...then apply to your project:
openspec update
```

`openspec update` regenerates the command files for your AI tool, so the new commands show up in chat.

## When to reach for what

- **Vague idea?** `/opsx:explore` first.
- **Clear, small feature?** `/opsx:propose`, then `/opsx:apply`.
- **Big or risky change?** Consider the expanded profile so you can create artifacts deliberately and `/opsx:verify` before archiving.
- **Several finished changes piling up?** `/opsx:bulk-archive` (expanded).

### `/opsx:ff` vs `/opsx:continue`

Both create planning artifacts in the expanded profile. The difference is pace.

| Situation | Use |
|-----------|-----|
| Clear requirements, ready to build | `/opsx:ff` |
| Exploring, want to review each step | `/opsx:continue` |
| Want to iterate on the proposal before specs | `/opsx:continue` |
| Time pressure, need to move fast | `/opsx:ff` |

**Rule of thumb:** if you can describe the full scope up front, fast-forward. If you are figuring it out as you go, continue one artifact at a time.

## Verify before you archive

`/opsx:verify` (expanded) checks your implementation against the change's artifacts across three dimensions. It reports issues but never blocks the archive.

```text
You: /opsx:verify

AI:  Verifying add-auth...

     COMPLETENESS
     ✓ All 12 tasks in tasks.md are checked
     ✓ All requirements in specs have corresponding code
     ⚠ Scenario "Session timeout after inactivity" not tested

     CORRECTNESS
     ✓ Implementation matches spec intent
     ✓ Edge cases from scenarios are handled

     COHERENCE
     ✓ Design decisions reflected in code structure
     ⚠ Design mentions "event-driven" but implementation uses polling

     SUMMARY  —  Critical: 0   Warnings: 2   Ready to archive: Yes
```

| Dimension | What it validates |
|-----------|-------------------|
| Completeness | All tasks done, all requirements implemented, scenarios covered |
| Correctness | Implementation matches spec intent, edge cases handled |
| Coherence | Design decisions reflected in code, patterns consistent |

## Update vs. start a new change

A question everyone hits: refine the change you have, or start a fresh one?

| Refine the existing change when... | Start a new change when... |
|---|---|
| Same intent, better execution | Intent fundamentally changed |
| Scope narrows (ship the MVP now, rest later) | Scope exploded into different work |
| Corrections from what you learned in the code | The original can be marked "done" on its own |

**Worked example — "add dark mode":**

- "System-preference detection is harder than expected" → **update** (same intent).
- "Let's ship the toggle now, add preferences later" → **update**, archive, then a new change for phase two.
- "Now we need a full theming system" → **new change** (scope exploded).

Keep each change focused on one logical unit of work. If you are doing "add feature X and also refactor Y," that is two changes.

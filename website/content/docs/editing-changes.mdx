**Every artifact in a change is just a Markdown file you can edit at any time.** There is no locked "planning phase," no approval gate, no special edit mode to enter. Want to change the proposal after you have started building? Open `proposal.md` and change it. Realized the design is wrong mid-implementation? Fix `design.md` and keep going. That is the whole answer, and it is by design.

This page is for the moment you think "wait, can I go back and change that?" Yes. Here is how, for each common case.

## Two ways to edit anything

<Cards>
  <Card title="Edit the file directly" icon="FileEdit">
    Artifacts are plain Markdown in `openspec/changes/<name>/`. Open `proposal.md`, `design.md`, `tasks.md`, or a delta spec and change it. Nothing else required.
  </Card>
  <Card title="Ask your AI to revise it" icon="Bot">
    In chat, just say what you want: "drop the caching idea and add a rate-limit section." The AI edits the artifact using the rest of the change as context.
  </Card>
</Cards>

Small wording tweak? Edit the file. Substantive rethink? Let the AI revise with full context.

## Updating the proposal (or specs) after you have started

Just update it. Same change, refined. The mental model: **artifacts are the live plan, not a signed contract.** The AI always works from their current contents, so editing them steers the work.

```text
You: Update design.md to use a background job instead of a synchronous call.

AI:  Updated design.md. The task list still fits; want me to continue applying?

You: /opsx:apply
```

There is no separate "update proposal" command because you do not need one. The file is the source of truth, and editing it — by hand or via the AI — *is* the update. On the default `core` commands, edit the artifact and run `/opsx:apply`; it reads the current files and builds against whatever they now say.

## Going back to review after implementing

You do not have to "go back," because you never left. Review, edit, and implementation are not sequential phases you are trapped in.

- **Re-examine the plan?** Open the artifacts, or run `openspec show <change>` in your terminal for a consolidated view.
- **Found something to change?** Edit the artifact (or ask the AI to), then continue.
- **Want a structured check?** Run `/opsx:verify` (expanded command). It reports completeness, correctness, and coherence without blocking anything.

## Reconciling manual code edits

You tweaked something in your editor, and now the code and the artifacts disagree. This happens constantly and it is fine. Bring them back in sync in whichever direction is true:

- **The code is now correct, the spec is stale.** Update the delta spec (and tasks, if relevant) to describe the behavior you actually shipped.
- **The spec is correct, the code drifted.** Keep building until the code matches the spec.

<Callout title="The principle">
At archive time, your specs become the truth of record. So before you archive, make the specs honest about what the code does. `/opsx:verify` is a fast way to surface mismatches — treat its output as a reconciliation to-do list.
</Callout>

## Update vs. start a new change

Short version: **update when it is the same work refined; start new when the intent fundamentally changed or the scope exploded into different work.**

| Situation | Move |
|-----------|------|
| Same goal, better approach | Update |
| Scope narrowing (ship the MVP now, more later) | Update, archive, then a new change for phase two |
| The problem itself changed ("add dark mode" became "build a full theming system") | New change |

## A note on tasks

`tasks.md` is a living checklist, not a frozen plan. As you implement, add tasks you discover, remove ones that turned out unnecessary, or reorder them. The AI checks items off during `/opsx:apply` and resumes from the first unchecked task if you come back later. Editing the list mid-flight is expected.

<Accordion title="Where do I type the commands?">
Two places. `openspec ...` commands run in your **terminal**. `/opsx:...` commands run in your **AI assistant's chat**. Mixing these up is the single most common early stumble — [How Commands Work](/docs/how-commands-work) explains the split.
</Accordion>

<Accordion title="Do I have to document my whole codebase before OpenSpec is useful?">
No. You write specs only for what you are about to change. Specs grow one change at a time and fill in naturally around the work you actually do. This is the whole point of the delta-based, brownfield-first design. See [Existing Projects](/docs/existing-projects).
</Accordion>

<Accordion title="What is a delta spec?">
Instead of rewriting an entire spec, a change describes only what is changing: which requirements are `ADDED`, `MODIFIED`, or `REMOVED`. When you archive the change, those deltas merge into your main specs. It is the trick that makes OpenSpec good at editing existing systems. See [Core Concepts](/docs/core-concepts).
</Accordion>

<Accordion title="Can I edit the proposal or design after I have started building?">
Yes, anytime. Every artifact is just a Markdown file. Edit it by hand or ask the AI to revise it, then keep going — nothing is locked. There is no separate "update" command because editing the file *is* the update. See [Editing a Change](/docs/editing-changes).
</Accordion>

<Accordion title="I already have a PRD / SRS / formal spec. Do I convert it?">
No bulk conversion. Treat existing docs as source material: point the AI at the relevant section when you start a change, and let it shape a focused delta. Your originals stay where they are as background. Forcing a one-time conversion tends to produce a large, stale spec nobody trusts.
</Accordion>

<Accordion title="Can I work on multiple changes at once?">
Yes. Each change is its own folder under `openspec/changes/`. Two changes can even touch the same spec file without conflicting, as long as they modify different requirements.
</Accordion>

<Accordion title="Do I have to use the explore command?">
No, it is optional — but it is the best habit to form. When you have a problem but not yet a plan, `/opsx:explore` reads your code, weighs options, and sharpens the idea before any artifact exists. When you already know exactly what you want, skip straight to `/opsx:propose`. See [Explore First](/docs/explore).
</Accordion>

<Accordion title="Which AI tools and models work best?">
OpenSpec works with 30+ assistants — Claude Code, Cursor, Codex, Windsurf, Gemini CLI, and more ([full list](/docs/reference/supported-tools)). It works best with high-reasoning models for both planning and implementation. A clean context window helps too: clear your context before starting implementation.
</Accordion>

<Accordion title="Should I commit the openspec/ folder to git?">
Yes. Your specs and archive are documentation of how your system behaves and why it changed. They belong in version control alongside the code they describe.
</Accordion>

<Accordion title="How do I keep OpenSpec up to date?">
Upgrade the package (`npm install -g @fission-ai/openspec@latest`), then run `openspec update` inside each project to regenerate the AI command files for the new version. See [Installation](/docs/installation).
</Accordion>

<Accordion title="Is it free? Does it collect data?">
OpenSpec is open source under the MIT license. It collects only anonymous command names and version numbers to understand usage — no arguments, paths, content, or personal data, and it is disabled automatically in CI. Opt out with `export OPENSPEC_TELEMETRY=0` or `export DO_NOT_TRACK=1`.
</Accordion>

<Accordion title="Where do I get help?">
Join the [Discord](https://discord.gg/YctCnvvshC), open a [GitHub issue](https://github.com/Fission-AI/OpenSpec/issues), or run `openspec feedback "your message"` from your terminal to file one directly.
</Accordion>

</Accordions>
